Student Age Analyzer
Seatwork | April 15, 2026 | BSIT 2-3
A Java program that takes a student's name and age as input, then categorizes them using a separate static method getAgeCategory() — demonstrating the use of methods in OOP.
  • Child: age 0–12
  • Teenager: age 13–19
  • Adult: age 20–59
  • Senior Citizen: age 60+
  • Invalid Age: negative or out-of-range values
public static String getAgeCategory(int age) {
  if (age >= 0 && age <= 12) { return "Child"; }
  else if (age >= 13 && age <= 19) { return "Teenager"; }
  else if (age >= 20 && age <= 59) { return "Adult"; }
  else if (age >= 60) { return "Senior Citizen"; }
  else { return "Invalid Age"; }
}
"The Student Age Analyzer was my first real encounter with a method that returns a meaningful value, and it immediately showed me why methods exist in the first place. Instead of cramming all the if-else logic into main(), I could call getAgeCategory() from anywhere and it would simply work. I also noticed how much cleaner and more readable the code became compared to doing everything inline. Writing the invalid age case made me think about edge cases seriously for the first time — what happens when someone enters -5 or 200? Thinking defensively like that felt like a genuine shift toward writing professional-quality code rather than code that only handles the expected input."

Activity 1 — Code Analysis (Variables)
OOP Activity | March 13, 2026 | BSIT 2-3
Analyzed a Java Account class to identify variable types, memory behavior (Stack vs Heap), and how object references work. This deepened my understanding of how Java manages memory. 
// Part A – Variable Identification
// Variable   | Data Type | Variable Type      | Stored In
// owner      | String    | Instance Variable   | Heap
// balance    | double    | Instance Variable   | Heap
// acc1       | Account   | Reference Variable  | Stack
// acc2       | Account   | Reference Variable  | Stack
// acc3       | Account   | Reference Variable  | Stack

// Part C – Output
// Miguel 15000.0
// Diana  388000.0
// Miguel 15000.0
  • Instance variables (owner, balance) are stored in the Heap
  • Reference variables (acc1, acc2, acc3) are stored in the Stack
  • When acc3 = acc1, no new object is created — both reference the same Heap object
  • Modifying acc3's owner changes the object shared by acc1 as well
"This activity genuinely changed how I think about variables in Java. Before this analysis, I treated all variables the same — just containers holding values. Understanding that reference variables only point to objects living in the Heap, and that multiple references can share the exact same object, finally explained bugs I had seen in earlier code but could never quite explain. The moment everything clicked was when I traced why changing acc3's owner also changed acc1's printed output: they were not two separate Miguels — they were the same object in memory. That mental model of Stack pointers pointing to Heap objects is something I now think about consciously every time I write object-based code."

Activity 3 — Basic ATM System
Java Program | BSIT 2-3 | Eclipse IDE
A menu-driven ATM simulation using a do-while loop and switch-case. Users can check balance, deposit, and withdraw — with input validation and a ₱2,000 withdrawal limit per transaction. 
double balance = 5000;
do {
  // Display menu options 1–4
  switch (choice) {
    case 1: // Check Balance
    case 2: balance += depositAmount; // Deposit
    case 3: // Withdraw — limit: ₱2,000 max
      if (withdrawalAmount > 2000)
        System.out.println("Withdrawal limit exceeded!");
      else balance -= withdrawalAmount;
    case 4: // Exit
  }
} while (choice != 4);
  • do-while ensures the menu displays at least once before checking exit condition
  • switch-case cleanly separates the four transaction types
  • Input validation prevents invalid deposits and over-limit withdrawals
"This activity clarified the practical difference between while and do-while in a way that textbook explanations never quite did. Because an ATM always needs to show its menu at least once before asking whether to continue, do-while was not just a stylistic choice — it was the correct choice for the problem. Enforcing the ₱2,000 withdrawal limit also pushed me to think about real-world constraints inside code, not just correct syntax. By the time I finished, I felt like I had built something with genuine purpose, not just an exercise — and that motivation made the debugging process feel worthwhile rather than frustrating."

Activity 5 — Personal Expense Tracker
Java Program | April 16, 2026 | BSIT 2-3
A modular Java application that uses static methods to organize code into logical units: displayTitle(), calculateTotalExpenses(), checkBudgetStatus(), and displaySummary(). Demonstrates the use of methods — a core OOP principle. 
public static double calculateTotalExpenses(
    double food, double transport, double other,
    double utilities, double entertainment) {
  return food + transport + other + entertainment + utilities;
}

public static String checkBudgetStatus(double total, double budget) {
  if (total > budget)
    return "⚠ WARNING: Budget Exceeded by PHP " + (total - budget);
  else
    return "✔ GOOD JOB: You are within your budget!";
}
  • Static methods allow code reuse without creating objects
  • Separating logic into methods makes the code modular and readable
  • printf() with format specifiers ensures clean, aligned output
  • Demonstrated budget analysis with conditional feedback messages
"Breaking a program into multiple methods taught me to think modularly — each method does one thing and does it well. This is the foundation of clean OOP design. What stood out most was how much easier debugging became: when the budget calculation was wrong, I only had to look inside calculateTotalExpenses() instead of hunting through dozens of lines in main(). That isolation of responsibility is exactly what makes large programs manageable. Designing the formatted summary output also pushed me to care about how users experience the program, not just whether it runs correctly — a distinction that I think separates good developers from great ones."
